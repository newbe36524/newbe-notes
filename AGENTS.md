# newbe-notes - Agent Configuration

## Project Purpose

`newbe-notes` 是个人知识库，但其定位不是项目内部文档库，而是**项目无关的经验总结仓库**。

这里沉淀的内容应尽量具备：

- 可复用
- 可迁移
- 可公开发布
- 可脱离具体项目单独阅读

结论是：默认把此仓库视为**面向互联网发布的公开内容源**，而不是私有工作备忘录。

## Core Principle

写入此仓库的内容，默认应满足一句话：

**即使明天直接公开发布，也不应暴露项目隐私、客户隐私、账户信息或内部实现细节。**

## Content Positioning

适合写入：

- 通用工程经验
- 通用运维经验
- 发布流程总结
- 工具使用经验
- 模板、检查单、排障思路
- 与具体业务弱绑定的方法论

不适合直接写入：

- 某个私有项目的完整内部方案
- 未公开的产品计划
- 客户信息
- 内网环境信息
- 凭据、密钥、Token、Cookie
- 带真实业务数据的排障记录
- 可直接定位到具体公司、仓库、机器、人员的敏感描述

若确需引用项目经验，必须先抽象、泛化、脱敏，再写入。

## Privacy And Sanitization Rules

### Default Rule

默认脱敏，默认泛化，默认最小披露。

若无法确定某信息是否敏感，按敏感处理，不写或改写。

### Must Remove Or Generalize

以下内容不得直接写入公开笔记：

- 私有仓库名、内部项目代号、未公开产品名
- 客户名、供应商名、合作方实名
- 真实姓名、邮箱、手机号、身份证、工号、账号
- 服务器公网 IP、内网 IP、域名、VPN 地址、跳板机信息
- Access Key、Secret、Token、Cookie、连接串、证书、License
- 数据库名、表名、真实业务数据、日志原文
- 内部截图、后台地址、监控地址、工单链接
- 未公开的架构图、流程图、组织信息

### Recommended Replacements

需要保留结构时，用通用占位表达：

- `项目 A`、`项目 B`
- `某单体应用`、`某前端站点`、`某 .NET 服务`
- `example.com`
- `192.0.2.10`
- `user@example.com`
- `<TOKEN>`
- `<INTERNAL_URL>`
- `<PRIVATE_REPO>`

### Example Handling

不推荐：

- “在 hagicode-mono 的某私有服务里，通过 10.0.3.12 连数据库……”

推荐：

- “在某 monorepo 服务中，通过内网数据库连接完成……”

重点是保留经验，不保留可追踪身份。

## Writing Guidelines

### Write For Reuse

优先写“方法”和“判断”，不要把内容绑死在单一项目上。

更推荐：

- 问题是什么
- 为什么这样做
- 方案如何落地
- 有哪些边界与风险
- 哪些部分可复用

少写：

- 某次临时对话式记录
- 只对单个项目成立的上下文
- 无法公开的决策背景

### Prefer Abstraction Over Chronicle

若来源于真实项目实践，应优先抽象为：

- 模式
- 检查单
- 模板
- 经验总结
- 反例与坑点

而不是逐日流水账。

### Public-Facing Tone

行文可简洁，但要保证外部读者能理解。

默认假设读者不知道你的项目背景，因此：

- 首次出现的术语要自解释
- 不依赖只有作者自己知道的上下文
- 不写“这个项目里那个服务”这类悬空指代

## Editing Existing Notes

在修改现有笔记时，Agent 应先做一次“公开发布审查”：

1. 是否出现了具体项目身份
2. 是否暴露了内部环境
3. 是否带出真实账号、地址、截图或日志
4. 是否存在未脱敏命令、配置、链接
5. 这篇文章是否仍然是“项目无关经验”

若答案有任一为“是”，应先脱敏或重写，再继续扩展内容。

## Directory Guidance

- `00-Index/`：导航与入口
- `10-Projects/`：可以记录项目相关线索，但仍应避免直接暴露隐私；若要公开沉淀，优先抽象后再写
- `20-Engineering/`：通用工程经验
- `30-Operations/`：通用运维、交付、发布经验
- `90-Templates/`：可复用模板
- `99-Inbox/`：临时收纳区；正式沉淀前也应进行脱敏审查

## Agent Behavior

当 AI 助手在此仓库工作时，应遵守：

1. 先判断内容是否适合公开发布
2. 优先输出通用经验，而非项目专属细节
3. 对所有可能识别真实项目的信息做脱敏或泛化
4. 不写入任何密钥、凭据、内部地址或真实业务数据
5. 若用户要求记录具体案例，默认先改写为匿名化经验
6. 若用户要求保留敏感细节，应主动提醒此仓库是公开导向仓库，不宜直接写入

## Preferred Output Style

- Markdown 为主
- 标题清楚
- 结构简洁
- 以经验、步骤、检查单、模板为主
- 示例默认使用匿名占位符

## Final Check

提交前，至少自查一次：

- 是否可直接公开
- 是否已脱敏
- 是否仍然保留复用价值
- 是否去除了对单一项目的强依赖

若不能同时满足，说明内容还不该进入此仓库。
